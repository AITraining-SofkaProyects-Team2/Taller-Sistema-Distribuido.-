## Propuesta de mejora de rendimiento para test TC-032

### Contexto

El archivo `src/types/Ticket.ts` define tres string union types: `IncidentType`, `Priority` y `TicketStatus`. Estos tipos son evaluados como referencias a strings en tiempo de ejecución.

### Propuesta

Reemplazar los string union types por `const enum`. Los `const enum` son **inlinados por el compilador de TypeScript** — cada referencia se sustituye por el valor literal directamente en el código generado, eliminando cualquier overhead de lookup en runtime.

**Implementación propuesta:**

```typescript
// Antes: string union (referencia en runtime)
export type IncidentType =
  | 'NO_SERVICE'
  | 'INTERMITTENT_SERVICE'
  | 'SLOW_CONNECTION'
  | 'ROUTER_ISSUE'
  | 'BILLING_QUESTION'
  | 'OTHER';

export type Priority = 'HIGH' | 'MEDIUM' | 'LOW' | 'PENDING';

export type TicketStatus = 'RECEIVED' | 'IN_PROGRESS';

// Después: const enum (inlinado en compilación, cero overhead runtime)
export const enum IncidentType {
  NO_SERVICE          = 'NO_SERVICE',
  INTERMITTENT_SERVICE = 'INTERMITTENT_SERVICE',
  SLOW_CONNECTION     = 'SLOW_CONNECTION',
  ROUTER_ISSUE        = 'ROUTER_ISSUE',
  BILLING_QUESTION    = 'BILLING_QUESTION',
  OTHER               = 'OTHER',
}

export const enum Priority {
  HIGH    = 'HIGH',
  MEDIUM  = 'MEDIUM',
  LOW     = 'LOW',
  PENDING = 'PENDING',
}

export const enum TicketStatus {
  RECEIVED    = 'RECEIVED',
  IN_PROGRESS = 'IN_PROGRESS',
}
```

### Beneficio

En un sistema de consultas de alta frecuencia (dashboard con muchos tickets), las comparaciones de tipo, prioridad y estado se resuelven como literales directos en lugar de referencias a strings en memoria. El beneficio escala proporcionalmente al volumen de tickets procesados y la frecuencia de las consultas.

### Consideraciones

- Requiere actualizar todos los puntos de uso para referenciar `Priority.HIGH` en lugar de `'HIGH'`, etc.
- Los `const enum` no son compatibles con `isolatedModules: true` — verificar configuración de `tsconfig.json` antes de aplicar.
- Impacto en serialización JSON: el valor en runtime sigue siendo el string literal, por lo que la API no se ve afectada.

## Propuesta de mejora de rendimiento para test TC-034

### Contexto

En `TicketQueryService.findByLineNumber`, el mensaje del error de validación interpola el valor recibido mediante template literal: `` `Invalid lineNumber: must be exactly 10 digits, got "${lineNumber}"` ``. Esto construye un nuevo string en el heap en cada llamada rechazada, generando presión innecesaria sobre el Garbage Collector en escenarios de alta carga (stress tests, validación masiva de inputs malformados).

### Propuesta

Extraer el mensaje de error como constante de módulo y lanzar siempre el mismo string estático, eliminando la allocación dinámica por rechazo:

```typescript
const LINE_NUMBER_REGEX = /^\d{10}$/;
const INVALID_LINE_NUMBER_ERROR = 'Invalid lineNumber: must be exactly 10 digits';

export class TicketQueryService {
  constructor(private readonly repository: ITicketRepository) {}

  async findByLineNumber(lineNumber: string): Promise<Ticket[]> {
    if (!LINE_NUMBER_REGEX.test(lineNumber)) {
      throw new Error(INVALID_LINE_NUMBER_ERROR);
    }
    return this.repository.findByLineNumber(lineNumber);
  }
}
```

### Beneficio

| Métrica | Antes | Después |
|---------|-------|---------|
| Allocaciones de string por rechazo | 1 nuevo string en heap | 0 (reutiliza constante) |
| Presión sobre GC en alta carga | proporcional al volumen de inválidos | constante |
| Compatibilidad con tests actuales | ✅ | ✅ (`rejects.toThrow()` no verifica mensaje) |

### Consideraciones

- La pérdida de información del valor inválido en el mensaje puede dificultar el diagnóstico en logs de producción. Evaluar si es preferible mantener el valor en un campo separado del error (ej. error cause o un custom error con propiedad `value`).
- Compatible sin cambios con el setup actual de Vitest ya que los tests solo verifican que se lanza un error, no el contenido del mensaje.
