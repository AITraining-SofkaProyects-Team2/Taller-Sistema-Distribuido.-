# Mejoras propuestas para HU-07 — Búsqueda por número de línea

---

## Propuesta de mejora de rendimiento para test TC-032

### Contexto

El archivo `src/types/Ticket.ts` define tres string union types: `IncidentType`, `Priority` y `TicketStatus`. Estos tipos son evaluados como referencias a strings en tiempo de ejecución.

### Propuesta

Reemplazar los string union types por `const enum`. Los `const enum` son **inlinados por el compilador de TypeScript** — cada referencia se sustituye por el valor literal directamente en el código generado, eliminando cualquier overhead de lookup en runtime.

**Implementación propuesta:**

```typescript
// Antes: string union (referencia en runtime)
export type IncidentType =
  | 'NO_SERVICE'
  | 'INTERMITTENT_SERVICE'
  | 'SLOW_CONNECTION'
  | 'ROUTER_ISSUE'
  | 'BILLING_QUESTION'
  | 'OTHER';

export type Priority = 'HIGH' | 'MEDIUM' | 'LOW' | 'PENDING';

export type TicketStatus = 'RECEIVED' | 'IN_PROGRESS';

// Después: const enum (inlinado en compilación, cero overhead runtime)
export const enum IncidentType {
  NO_SERVICE          = 'NO_SERVICE',
  INTERMITTENT_SERVICE = 'INTERMITTENT_SERVICE',
  SLOW_CONNECTION     = 'SLOW_CONNECTION',
  ROUTER_ISSUE        = 'ROUTER_ISSUE',
  BILLING_QUESTION    = 'BILLING_QUESTION',
  OTHER               = 'OTHER',
}

export const enum Priority {
  HIGH    = 'HIGH',
  MEDIUM  = 'MEDIUM',
  LOW     = 'LOW',
  PENDING = 'PENDING',
}

export const enum TicketStatus {
  RECEIVED    = 'RECEIVED',
  IN_PROGRESS = 'IN_PROGRESS',
}
```

### Beneficio

En un sistema de consultas de alta frecuencia (dashboard con muchos tickets), las comparaciones de tipo, prioridad y estado se resuelven como literales directos en lugar de referencias a strings en memoria. El beneficio escala proporcionalmente al volumen de tickets procesados y la frecuencia de las consultas.

### Consideraciones

- Requiere actualizar todos los puntos de uso para referenciar `Priority.HIGH` en lugar de `'HIGH'`, etc.
- Los `const enum` no son compatibles con `isolatedModules: true` — verificar configuración de `tsconfig.json` antes de aplicar.
- Impacto en serialización JSON: el valor en runtime sigue siendo el string literal, por lo que la API no se ve afectada.

## Propuesta de mejora de rendimiento para test TC-032 y TC-033

### Contexto

En la suite `HU-07.test.ts`, cada `it(...)` realiza una llamada `await` independiente a `service.findByLineNumber(...)`. En TC-032 hay **10 invocaciones asíncronas** al mock y en TC-033 hay **3**, todas devolviendo el mismo resultado. Esto genera overhead innecesario de resolución de promesas por cada aserción.

### Propuesta

Extraer la única llamada al servicio a un `beforeAll`, guardar el resultado en una variable compartida y convertir todos los `it` a funciones síncronas que operen sobre ese resultado ya resuelto.

```typescript
describe('Given existen 3 tickets con lineNumber "0991234567" en el sistema', () => {
  let result: Ticket[];

  beforeAll(async () => {
    result = await service.findByLineNumber(VALID_LINE_NUMBER);
  });

  it('Then retorna un resultado no nulo', () => {
    expect(result).not.toBeNull();
  });

  it('Then retorna un arreglo', () => {
    expect(Array.isArray(result)).toBe(true);
  });

  it('Then retorna exactamente 3 tickets', () => {
    expect(result).toHaveLength(3);
  });

  it('Then todos los tickets tienen lineNumber igual a "0991234567"', () => {
    for (const ticket of result) {
      expect(ticket.lineNumber).toBe(VALID_LINE_NUMBER);
    }
  });

  // ... resto de its síncronos
});
```

### Beneficio

| Métrica | Antes | Después |
|---------|-------|---------|
| Llamadas `await` en TC-032 | 10 | 1 |
| Llamadas `await` en TC-033 | 3 | 1 |
| Tests marcados como `async` | 13 | 2 (`beforeAll`) |
| Overhead de resolución de promesas | alto | mínimo |

Cada resolución de promesa implica una microtarea en el event loop de Node.js. Al consolidarlas en un único `beforeAll`, se eliminan 12 ciclos del scheduler por ejecución de la suite, lo que escala en pipelines de CI con muchas suites paralelas.

### Consideraciones

- `beforeAll` comparte estado entre `it`s del mismo `describe` — asegurarse de que ningún `it` mute el arreglo `result`. Si algún test necesita mutarlo, usar `result.slice()` o `structuredClone(result)`.
- El mock de `findByLineNumber` sigue siendo invocado UNA vez por suite (verificable con `toHaveBeenCalledOnce`), lo que mantiene la semántica del test original.
- Compatible sin cambios con Vitest y el setup actual del proyecto.
