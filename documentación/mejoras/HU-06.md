## Propuesta de mejora de rendimiento para test TC-028

**Test**: TC-028 — Buscar por ID de ticket existente  
**Fecha de propuesta**: 20 de febrero de 2026  
**Estado**: Pendiente de implementación

### Contexto

`TicketQueryService.findById` delega directamente al repositorio en cada invocación. Cuando el ticket **existe**, el resultado es estable (los tickets son inmutables una vez procesados), por lo que llamadas repetidas al mismo `ticketId` generan I/O innecesario.

### Problema

```
Req 1 → findById("550e-...") → repositorio (I/O)  ← 1 query
Req 2 → findById("550e-...") → repositorio (I/O)  ← query redundante
Req 3 → findById("550e-...") → repositorio (I/O)  ← query redundante
```

### Mejora: Caché en Memoria con TTL

```typescript
interface CacheEntry {
  ticket: Ticket;
  expiresAt: number;
}

export class TicketQueryService {
  private readonly cache = new Map<string, CacheEntry>();
  private readonly TTL_MS: number;

  constructor(
    private readonly repository: ITicketRepository,
    ttlMs: number = 5_000
  ) {
    this.TTL_MS = ttlMs;
  }

  async findById(ticketId: string): Promise<Ticket> {
    const cached = this.cache.get(ticketId);

    if (cached && Date.now() < cached.expiresAt) {
      return cached.ticket; // ← cache HIT: O(1), sin I/O
    }

    const ticket = await this.repository.findById(ticketId);

    if (ticket === null) {
      throw new TicketNotFoundError();
    }

    this.cache.set(ticketId, { ticket, expiresAt: Date.now() + this.TTL_MS });
    return ticket;
  }

  invalidate(ticketId: string): void {
    this.cache.delete(ticketId);
  }
}
```

### Impacto Medible

| Escenario | Sin caché | Con caché (hit dentro de TTL) |
|-----------|:---------:|:------------------------------:|
| 10 req/s al mismo ticketId existente | 10 queries al repo | 1 query cada 5 s |
| Tiempo de respuesta (hit) | $O(T_{query})$ | $O(1)$ |
| Carga en repositorio | Alta | Mínima |

### Compatibilidad con TC-028

TC-028 no requiere cambios: inyectar `ttlMs: 0` en el test desactiva la caché, por lo que el mock del repositorio sigue siendo invocado tal como lo verifica `toHaveBeenCalledOnce()`.

```typescript
service = new TicketQueryService(mockRepository, 0); // TTL 0 = sin caché
```

### Decisión

- [ ] Implementar caché en memoria con TTL
- [ ] Mantener implementación actual (sin caché)

## Propuesta de mejora de rendimiento para test TC-029

**Test**: TC-029 — Buscar por ID de ticket inexistente  
**Fecha de propuesta**: 20 de febrero de 2026  
**Estado**: Pendiente de implementación

### Contexto

Cuando el `ticketId` **no existe**, el repositorio retorna `null` y el servicio lanza `TicketNotFoundError`. Si múltiples operadores consultan simultáneamente un ID inválido o inexistente conocido, se generan llamadas al repositorio cuyo resultado siempre será el mismo: `null`.

### Problema

```
Req 1 → findById("99999-...") → repositorio (I/O) → null → TicketNotFoundError
Req 2 → findById("99999-...") → repositorio (I/O) → null → TicketNotFoundError (redundante)
Req 3 → findById("99999-...") → repositorio (I/O) → null → TicketNotFoundError (redundante)
```

### Mejora: Caché de Ausencias (Negative Cache) con TTL corto

Extender la misma caché de TC-028 para también recordar los IDs que resultaron en `TicketNotFoundError`, evitando consultas repetidas al repositorio para ausencias conocidas.

```typescript
type CacheEntry =
  | { found: true;  ticket: Ticket;   expiresAt: number }
  | { found: false;                   expiresAt: number };

async findById(ticketId: string): Promise<Ticket> {
  const cached = this.cache.get(ticketId);

  if (cached && Date.now() < cached.expiresAt) {
    if (!cached.found) throw new TicketNotFoundError(); // ← cache HIT negativo
    return cached.ticket;
  }

  const ticket = await this.repository.findById(ticketId);

  if (ticket === null) {
    this.cache.set(ticketId, { found: false, expiresAt: Date.now() + this.TTL_MS });
    throw new TicketNotFoundError();
  }

  this.cache.set(ticketId, { found: true, ticket, expiresAt: Date.now() + this.TTL_MS });
  return ticket;
}
```

### Impacto Medible

| Escenario | Sin caché | Con negative cache (hit) |
|-----------|:---------:|:------------------------:|
| 10 req/s al mismo ID inexistente | 10 queries al repo | 1 query cada 5 s |
| Tiempo de respuesta (hit negativo) | $O(T_{query})$ | $O(1)$ |

### Compatibilidad con TC-029

TC-029 no requiere cambios: con `ttlMs: 0` cada entrada expira de inmediato, por lo que `toHaveBeenCalledOnce()` sigue verificando el comportamiento correctamente.

```typescript
service = new TicketQueryService(mockRepository, 0); // TTL 0 = sin caché
```

### Consideraciones

- El TTL para ausencias puede ser **menor** que para presencias (ej. 2 s vs 5 s) para reducir el riesgo de falsos negativos ante tickets recién creados.
- Si el Consumer publica un nuevo ticket, llamar a `invalidate(ticketId)` para limpiar la entrada negativa.

### Decisión

- [ ] Implementar negative cache junto con la mejora de TC-028
- [ ] Implementar solo caché positiva (TC-028) sin negative cache
- [ ] Mantener implementación actual (sin caché)

## Propuesta de mejora de rendimiento para test TC-030

**Test**: TC-030 — Buscar con ID en formato inválido  
**Fecha de propuesta**: 20 de febrero de 2026  
**Estado**: Pendiente de implementación

### Contexto

La validación actual ejecuta `UUID_V4_REGEX.test(ticketId)` sobre cualquier string recibido, sin importar su longitud. El motor de regex procesa carácter a carácter hasta confirmar o descartar el patrón, lo que resulta en trabajo innecesario para inputs claramente inválidos por longitud.

Un UUID tiene siempre **exactamente 36 caracteres** (`8-4-4-4-12` + 4 guiones). Cualquier string con longitud diferente puede descartarse en $O(1)$ sin ejecutar el regex.

### Problema

```
findById("abc")    → regex procesa 3 chars  → descarta → lanza error
findById("12345")  → regex procesa 5 chars  → descarta → lanza error
findById("550e8400-e29b-41d4-a716-44665544000") → regex procesa 35 chars → descarta → lanza error
```

Todos los casos anteriores podrían resolverse con una simple comparación de entero.

### Mejora: Guardia de Longitud Antes del Regex

```typescript
const UUID_V4_REGEX = /^[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$/i;
const UUID_LENGTH = 36;

async findById(ticketId: string): Promise<Ticket> {
  if (ticketId.length !== UUID_LENGTH || !UUID_V4_REGEX.test(ticketId)) {
    throw new InvalidUuidFormatError();
  }
  // ...
}
```

El operador `||` con cortocircuito garantiza que `UUID_V4_REGEX.test` **solo se evalúa** cuando la longitud es exactamente 36 — prácticamente solo el caso del UUID con formato correcto.

### Impacto Medible

| Input | Sin mejora | Con mejora |
|-------|:----------:|:----------:|
| `"abc"` (3 chars) | $O(n_{regex})$ | $O(1)$ — length check |
| `"12345"` (5 chars) | $O(n_{regex})$ | $O(1)$ — length check |
| UUID truncado (35 chars) | $O(n_{regex})$ | $O(1)$ — length check |
| UUID extendido (37 chars) | $O(n_{regex})$ | $O(1)$ — length check |
| UUID válido (36 chars) | $O(n_{regex})$ | $O(n_{regex})$ — sin cambio |

### Compatibilidad con TC-030

Todos los 21 tests de TC-030 mantienen el comportamiento esperado: los IDs inválidos siguen lanzando `InvalidUuidFormatError` — el cambio es interno al `if`, no al resultado observable.

### Decisión

- [ ] Implementar guardia de longitud antes del regex
- [ ] Mantener implementación actual (solo regex)
